# 设计决策

从当前代码看，这个项目做了几项非常明确的取舍，而且这些取舍互相强化。

## 1. 用 Python 标准库做后端

`session-manager.py` 直接基于 `http.server` 和 `BaseHTTPRequestHandler`。这意味着：

- 启动成本低
- 网页端零额外依赖
- 分发更简单

代价是没有现成的路由、中间件和测试生态，需要靠文件内组织保持可读性。

## 2. 前端保持零构建

选择 `index.html + app.js + style.css`，而不是 React / Vue 一类方案，显然是为了把门槛压到最低。对一个本地管理面板来说，这个选择很合理，因为真正复杂的是数据整理，不是页面路由。

## 3. 桌面端复用网页端后端

`session-manager-desktop.py` 没有单独实现业务，只是加载并嵌入同一个服务模块。这避免了 Web 与 Desktop 两套逻辑分叉，是整个仓库最重要的“不要重复自己”决策。

## 4. 打包入口统一

平台脚本只是包装，真正的逻辑全部集中在 `build-desktop.py`。这让 CI、本地构建和文档说明能围绕同一份脚本收敛，而不是为每个平台维护独立流程。

## 5. 本地数据优先于远程服务

所有操作都围绕本地 `sessions` 文件夹、浏览器、终端和桌面窗口展开。项目没有云同步、用户系统或服务端存储，安全假设和功能边界因此都更清楚。

看这些决策的代价，请继续看 [容易踩坑的地方](pitfalls.md)。
