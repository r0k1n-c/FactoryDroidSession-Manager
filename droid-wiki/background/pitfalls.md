# 容易踩坑的地方

这个项目实现不复杂，但有几类很容易在后续演进时放大的问题。

## 1. 全量扫描偏多

`scan_sessions()` 会遍历 sessions 目录，`search_sessions_by_message()` 也会重新逐条扫描 `.jsonl`。当前规模下可能没问题，但会话量一大，刷新和搜索都可能明显变慢。

## 2. 主逻辑集中在大文件里

`session-manager.py` 和 `frontend/assets/app.js` 都已经接近 750 行。继续在这两个文件里叠功能，后续维护成本会明显上升。

## 3. 删除操作不可恢复

`delete_session()` 直接 `os.remove()`，没有回收站、软删除或备份。对“本地工具”来说可以接受，但如果用户误点，恢复成本很高。

## 4. 当前 checkout 没有 git 元数据

这不影响运行，但会影响很多工程能力：没法做贡献者分析、时间线重建、churn 热点统计，也不利于正式协作。

## 5. 测试覆盖面很窄

桌面自检只能证明服务能起来，不能证明前端交互、删除逻辑或搜索逻辑一定正确。只要需求继续扩展，这会越来越成为约束。

这些问题已经汇总在 [清理机会](../cleanup-opportunities.md) 里，后续重构可以从那里开始排优先级。
