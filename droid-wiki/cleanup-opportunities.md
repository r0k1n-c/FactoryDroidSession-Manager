# 清理机会

当前仓库没有大量死代码或 TODO 堆积，但还是有几处很明确的维护热点。如果接下来要继续加功能，最好先给这些地方留出重构预算。

## 1. 两个超大主文件

| File | Lines | Why it is a hotspot |
|---|---:|---|
| `session-manager.py` | 752 | 同时承载目录发现、API、CLI、删除与启动逻辑 |
| `frontend/assets/app.js` | 743 | 同时承载状态管理、i18n、主题、详情、分析图表和清理动作 |

这两个文件都已经承担太多责任。最自然的拆分方向是：

- 后端：把目录发现、CLI、HTTP handler 拆成多个模块
- 前端：把列表、详情、分析图表拆成多个 JS 文件

## 2. 全量扫描与全文搜索成本

- `scan_sessions()` 每次都会全量扫描 workspace
- `search_sessions_by_message()` 每次都会重新遍历所有会话消息

如果未来 sessions 数量继续增长，优先考虑：

- 缓存会话摘要
- 增量刷新
- 预建简单索引

## 3. 自动化测试缺口

当前只有：

- `session-manager-desktop.py --self-test`

缺少：

- API 回归测试
- 删除动作测试
- 前端 smoke test
- 针对不同目录结构的样例数据测试

## 4. 文档重叠

`README.md` 与 `项目介绍.md` 有明显重叠。如果两份文档继续并行维护，后续很容易漂移。更稳妥的方式是明确：

- 一份对外说明
- 一份内部背景说明

## 5. 缺少完整仓库元数据

当前 checkout 没有 `.git`，这让很多工程性文档只能写成“快照说明”。如果这是长期工作目录，建议保留完整 git 元数据，便于后续维护 wiki、贡献者信息和构建演化记录。
